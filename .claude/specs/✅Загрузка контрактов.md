**Назначение раздела:** Инструмент для пакетного добавления десятков или сотен контрактов в базу данных платформы путем прямой синхронизации с внешней рабочей таблицей (Google Sheets) с последующей системной превалидацией данных.

## ЧАСТЬ 1. ИНТЕРФЕЙС И ИМПОРТ ДАННЫХ (БЕЗ ФАЙЛОВ)

**1.1. Триггер запуска** В разделе располагается основная кнопка: Загрузить контракты.

**1.2. Механика импорта (Интеграция)** Вместо вызова окна выбора файла на компьютере, система работает по прямой связке:

1. При нажатии на Загрузить контракты бэкенд платформы по API обращается к заранее привязанной мастер-таблице (ссылка на которую настроена в системе).
2. Система считывает все заполненные строки из таблицы.
3. Данные моментально переносятся в интерфейс платформы в виде предварительного реестра (таблицы) для проверки.

**1.3. Элементы управления предварительным реестром**

- Основная таблица со всеми вытянутыми колонками (Номер, ID клиента, ID консультанта, Продукт, Сумма, Даты и т.д.).
- Кнопка: Удалить все контракты (позволяет очистить буферную зону, если данные подтянулись некорректно, чтобы начать заново).

---

## ЧАСТЬ 2. СИСТЕМА ПРЕВАЛИДАЦИИ И ОБРАБОТКИ ОШИБОК

Самый важный этап работы модуля. Данные, подтянутые из таблицы, еще **не сохранены** в основную базу. Они находятся в буферной зоне, где система прогоняет их через Матрицу валидации.

**2.1. Индикация ошибок (UI)**

- Если строка прошла все проверки (уникальный номер, корректные ID, правильные форматы дат) — она отображается в стандартном виде.
- Если в строке найдена системная ошибка, слева от нее появляется индикатор — **Красный треугольник с восклицательным знаком**.
- **Тултип (Подсказка):** При наведении курсора на красный треугольник система должна выводить конкретный текст ошибки (например: *«Контракт с таким номером уже существует» *или *«Некорректный ID клиента»*.

**2.2. Жесткая блокировка сохранения**

- Кнопка финального сохранения **отсутствует (скрыта)** до тех пор, пока в списке есть хотя бы одна строка с ошибкой (красным треугольником). Это исключает попадание «грязных» данных в БД.

---

## ЧАСТЬ 3. РУЧНОЕ РЕДАКТИРОВАНИЕ И ФИНАЛИЗАЦИЯ

Если система нашла ошибки, администратор должен их исправить, не возвращаясь в исходную Google-таблицу.

**3.1. Исправление ошибок (Inline editing)** В конце каждой строки (и корректной, и с ошибкой) есть иконки управления:

- **Карандаш (Редактировать):** Открывает стандартное модальное окно контракта (как в Менеджере контрактов). Администратор может вручную исправить проблемное поле (например, изменить задублированный номер). При нажатии Сохранить внутри модального окна, строка обновляется и система прогоняет ее проверку заново. Если ошибка устранена — красный треугольник исчезает.
- **Корзина (Удалить):** Позволяет точечно удалить проблемную строку из буферного списка импорта, если ее невозможно или не нужно исправлять.

**3.2. Успешный сценарий (Happy Path)** Как только в предварительном реестре не остается ни одного красного треугольника (все ошибки исправлены или проблемные строки удалены), интерфейс обновляется:

- Появляется большая зеленая кнопка: Сохранить заполненные контракты.

**3.3. Фиксация в БД** При нажатии на эту кнопку:

1. Данные из буферной зоны переносятся в основную базу данных платформы.
2. Контракты появляются в общем разделе «Менеджер контрактов».
3. Экран массовой загрузки очищается, система готова к следующему импорту.
