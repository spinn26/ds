## Ключевые константы

- **Балл:** `1 балл = 100 ₽`. Revenue → points: `/100`. Commission points → rubles: `×100`.
- **НДС** хранится в таблице `vat` с колонками `dateFrom` / `dateTo`. Сейчас действует `5%`; формула `amountNoVat = amountRub / (1 + vat%/100)` эквивалентна `×100/105` при 5%.
- **Старт:** новый партнёр регистрируется как **Start (уровень 1)**, НГП = 0, закреплён за наставником (`consultant.inviter`).
- **Активация:** порог `500 ЛП`, срок `90 дней` от регистрации (см. `App\Enums\PartnerActivity`).

## Формулы

### Экономика сделки
```
Доход ДС            = сумма × %ДС
Доход ДС без НДС    = Доход ДС / (1 + vat%/100)          // при vat=5% это /1.05 = ×100/105
Доход ДС (USD→RUB)  = сумма × %ДС × курс USD(дата_заведения)
Прибыль компании    = Доход ДС без НДС − Σ(комиссии партнёров по цепочке)
```

### Вознаграждение
```
ЛП-комиссия(партнёр)     = объём_в_баллах × % партнёра
ГП-комиссия(наставник)   = объём_в_баллах × (% наставника − % нижестоящего)     // clamp в 0 если <0
```

### Штрафы (применяются монотонно, порядок строг)
```
# Отрыв — только для уровней 6-10 (status_levels.otrif=70, остальные=0)
if одна_ветка / ГП > status_levels.otrif/100 (и уровень ≥ TOP FC):
    commission(ветки) *= 0.5        // баллы-объём остаются 100%

# ОП — с уровня Expert (status_levels.mandatoryGP>0)
if ГП_факт < status_levels.mandatoryGP (и уровень ≥ Expert):
    Σ(ГП-комиссий) *= 0.8           // ЛП-комиссия не трогается

# Комбо: сначала отрыв (per-branch ×0.5), потом ОП (×0.8 к итогу ГП).
```

### Лидерский пул (TOP FC → Co-founder)
```
fund(уровень)   = выручка_ДС_без_НДС_за_месяц × 1%
share(уровень)  = fund / номинальная_численность_уровня    // включая неподтверждённых

# Матрёшка: партнёр уровня L получает share(L) + share(L-1) + ... + share(TOP FC).
# Форфейтнутые доли НЕ перераспределяются — остаются компании (спец. §6.5).
```

### Реестр выплат
```
Итого начислено   = Сальдо + Начислено + Прочие + Пул
Остаток           = Итого начислено − Оплачено
                    → сальдо следующего месяца
```

## Операционная модель пула (актуальная, 2026-04-20)

По спеке [Пул](./✅Пул.md) пул **не автоматизирован полностью** — есть шаг ручной модерации:

1. Автомат: за месяц система собирает всех партнёров уровня ≥ 6 в список «Участники пула на `[Месяц Год]`».
2. У каждого стоит чекбокс «Участвует» (по умолчанию ✓).
3. Оператор вручную снимает галочки у нарушителей (ОП-недобор, отрыв >90%, любые другие основания).
4. По нажатию «Рассчитать пул» система считает `share` на уровень и раздаёт по матрёшке. Форфейтнутые доли остаются компании.

То есть **порог отрыва 90% для пула — это подсказка оператору, не автоматический фильтр**. Отличается от обычного отрыва 70% (§5.1), который применяется системно к комиссиям.

## Инварианты, которые нельзя нарушить

1. **Закрытые периоды заморожены.** Если месяц закрыт (transaction.period = серый индикатор в [Комиссии](./✅Комиссии%20.md)), ни одна запись `commission` / `transaction` / `qualificationLog` / `poolLog` за этот месяц **не правится**. Исправления идут только через «Прочие начисления».
2. **Ручной override статуса пишется в аудит-лог.** Старое значение, новое значение, ID сотрудника, комментарий (см. [Статусы партнеров](./✅Статусы%20партнеров.md) ч. 3).
3. **«Прочие начисления» обратимы.** Удаление записи = реверсивная транзакция на балансе (см. [Прочие начисления](./✅Прочие%20начисления.md) ч. 2 п. 4).
4. **НДС и курс валют применяются по дате транзакции.** Изменение ставки пересчитывает только открытый период.
