## 1. Пользовательский интерфейс (UI)

Интерфейс представляет собой лаконичный блок настройки импорта (как показано на макете):

- **Выпадающий список «Выбрать поставщика»:** Специалист выбирает нужного поставщика из системного справочника (см листы таблицы импорта). Этот выбор определяет, с какого листа/источника система будет забирать данные.
- **Выпадающий список «Выбрать валюту»:** Появляется/активен, если выбранный поставщик поддерживает мульти-валютные расчеты. Позволяет отфильтровать загрузку только для конкретной валюты (например, RUB, USD, EUR).
- **Кнопка «Импортировать транзакции»:** Запускает системный алгоритм парсинга, сопоставления и расчетов.

## 2. Входные данные (Структура импортируемой таблицы)

Система ожидает получить данные в виде таблицы (где каждый лист соответствует поставщику) со набором колонок зависимо от поставщика (см. листы таблицы импорта), но в основном это:

- Номер контракта (**Ключевое поле для сопоставления**)
- Сумма контракта 
- Валюта контракта (USD, EUR, GBR, RUB и т.д.)
- Дата (дата проведения транзакции/оплаты)
- уровень ДС - опцыонально, где есть
- год контракта - опцыонально, где есть
- свойство продукта - опцыонально, где есть

## 3. Пошаговый алгоритм работы системы (Бэкенд)

**Шаг 1. Инициализация и Парсинг** Специалист выбирает параметры (Поставщик + Валюта) и нажимает кнопку импорта. Система обращается к соответствующему массиву данных (листу таблицы) и начинает построчную обработку записей, соответствующих выбранной валюте.

**Шаг 2. Поиск и сопоставление (Матчинг)** Для каждой строки система берет значение из колонки **Номер контракта** и ищет полное совпадение в собственной базе данных (БД) платформы нужный контракт.

- *Успех:* Если контракт с таким номером найден, система понимает, какому клиенту и какому партнёру (консультанту) принадлежит эта продажа.
- *Ошибка (Исключение):* Если контракт не найден, строка помечается ошибкой импорта (рекомендуется выводить специалисту лог ошибок после завершения загрузки для ручного разбора).

**Шаг 3. Создание системной транзакции** При успешном нахождении контракта система автоматически создает сущность «Транзакция» и привязывает её к найденному контракту. В транзакцию записываются данные из таблицы: дата, суммы (в валюте и рублях), процент поставщика.

**Шаг 4. Каскадный расчет баллов и комиссий (Запуск MLM-движка)** Как только транзакция создана, система автоматически инициирует скрипты расчетов (см. расчет комиссионных):

1. **Для прямого партнера (Консультанта):**
   - Рассчитывается Личный Объем (ЛП) в баллах на основе Суммы контракта в рублях и формулы расчета баллов для конкретного продукта.
   - Рассчитывается комиссионное вознаграждение в баллах (ЛП * % текущей квалификации партнера).
   - Баллы конвертируются в рубли (1 балл = 100 рублей) и зачисляются на баланс партнера.
2. **Для вышестоящих наставников (Структуры вверх):**
   - Система обращается к массиву upperLevels (списку всех наставников над этим партнером).
   - Вычисляет разницу процентов квалификаций (маржинальную разницу) между текущим наставником и нижестоящим партнером.
   - Зачисляет вышестоящим наставникам Групповой Объем (ГП) и рассчитывает их комиссию от групповых продаж. 
     
     ***Проверка правила «Отрыва» и Проверка «Выполнения плана продаж» и расчет по ним - должны происходить по команде от сотрудника, когда тот внесет все транзакции за период. - !можно делать в моменте если разрешает система. !***
     *Подробное описание расчетов и проверок правил см. Логика расчетов коммисий партнеров*

**Результат операции:** Специалист видит системное уведомление об успешном завершении импорта (например, «Успешно импортировано: 45 транзакций. Ошибок: 0»). Балансы и объемы в личных кабинетах партнёров обновляются сразу.
